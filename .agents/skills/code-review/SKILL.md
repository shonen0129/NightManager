# Code Review スキル

## 目的

リリース前の最終監査として、コードベース全体を体系的に調査し、実害のある問題を網羅的に洗い出す。

## 基本方針

- ソースコードは修正せず、レビューと検証のみを行う
- リポジトリ全体を対象とする
- 件数に上限を設けない。一方で根拠の弱い問題を作らない
- コードスタイルや好みの指摘は避け、動作・信頼性・安全性・保守性に実質的な影響があるもののみ

## 変更差分レビュー時の追加チェックリスト

リリース前の全体監査ではなく「特定の変更に対するレビュー」を求められた場合、以下を Step 1 の前に必ず実行する:

1. **変更が既存実装と重複していないか**: `code_search` / `grep_search` で、追加した機能と同等の処理が既に別箇所に存在しないか確認する
2. **変更が実際に必要か**: デフォルト値のみ使用されるパラメータ等、実動作に影響しない変更は「変更しない」が正解の場合がある
3. **外部APIのパラメータマッピングが仕様書と一致するか**: `docs/api/` 配下の仕様書を確認せず推測でマッピングを書いていないか
4. **変更前後の実動作の差分**: どの入力条件で挙動が変わるかを明示する。変わらないなら変更の意義を疑う

## 調査手順

### Step 1: リポジトリ全体の把握（目安: 15分）

以下を整理する:
- ディレクトリ構成と主要コンポーネント
- エントリーポイント（CLI、本番スクリプト、バッチ）
- データの入出力経路（yfinance → preprocessor → df_exec → signal → portfolio → execution）
- 外部サービスとの連携（証券API、yfinance）
- 非同期処理、ジョブ、スケジューラ
- ビルド、テスト、デプロイ方法

主要な処理について入力から出力までの実行経路を追跡する。個別ファイルを孤立して読まず、呼び出し元と呼び出し先を含めて確認する。

最初に確認すべきファイル:
- `AGENTS.md` — 不変条件・既知の落とし穴・よく使うコマンド
- `docs/ARCHITECTURE.md` — アーキテクチャ詳細・リファクタリング履歴
- `configs/production/production.yaml` — 本番config（正本）
- `requirements.txt`, `pyproject.toml` — 依存関係
- `tests/` — テストコード

### Step 2: 機械的検証（目安: 25分）

以下のコマンドを順に実行し、失敗した場合はその失敗自体を分析対象とする:

```bash
# 構文チェック
python3 _check_syntax.py

# Lint
python3 -m ruff check src/leadlag/ --select E,F,W --ignore E501

# テスト（並列・推奨、要 pytest-xdist）
bash scripts/run_tests_parallel.sh

# テスト（直列フォールバック）
python3 -m pytest tests/ -v --timeout=300
```

レビューのためにソースコードを書き換えてテストを通すことは禁止する。

### Step 3: 観点別レビュー（目安: 40分）

各観点を独立したレビュー工程として確認する。該当しない項目は「該当なし」と明記する。

**A. ドメイン固有リスク（最優先）**
1. ルックアヘッドリーク: ローリング統計・ベータ・PITビニングが strictly historical か
2. ベースライン期間の分離: 事前分布・基準相関が2010-2014固定か
3. 市場中立制約: net exposure ±0.05、gross ≤ 2.0 が守られているか
4. 数値安定性: ゼロ除算、NaN/Inf伝播、特異行列、PSD性の保証
5. フォールバック挙動: データ欠損・監査失敗時の挙動が安全側か
6. コンプライアンス監査: 監査項目が無効化されていないか

**B. 一般的品質リスク**
7. 通常系のロジックエラー
8. 境界値、空値、NaN、未定義値
9. エラー処理と例外処理（except Exception の濫用、静かな握り潰し）
10. 状態管理とデータ整合性（グローバルキャッシュの汚染・未クリア）
11. 並行処理、競合状態、スレッド安全性
12. 再試行、重複実行、冪等性
13. 入力検証、インジェクション、情報漏えい
14. 秘密情報、ログ、個人情報の取り扱い
15. パフォーマンス、無駄なI/O、メモリリーク
16. タイムアウト、ハング、異常終了
17. 設定値、環境変数、環境差異
18. 依存関係、バージョン、非推奨API
19. テスト不足と誤ったテスト
20. 重複実装、到達不能コード、古い実装
21. 将来の変更で不具合を誘発しやすい構造

### Step 4: 並行監査

`code_search` ツールを使用して、独立性の高い観点を並行調査する:
- アーキテクチャ・データフロー・仕様
- ロジック・例外処理・境界値
- リーク・コンプライアンス・フォールバック
- パフォーマンス・リソース管理

最終的な重要度判断と報告は自分自身が行う。

### Step 5: 反証と再走査（目安: 20分）

最初の問題一覧を「まだ不完全である」と仮定し、2回目のレビューを行う:

- 確認が薄かったディレクトリ・ファイル
- 指摘された問題と類似する実装
- 同じ関数やデータを利用する別経路
- テストが通っていても実運用で失敗する条件
- 正常系しか確認されていない処理
- 暗黙の前提に依存している処理
- 複数機能をまたぐ組み合わせ上の問題
- 障害発生時や部分的失敗時の挙動

新しい問題が見つかった場合は、その周辺をさらに調査する。

## 指摘の採用条件

各問題は実際のコードで裏付ける。以下を説明できるもののみ正式な指摘とする:

- どのコードが原因か（ファイル・行番号）
- どの条件で発生するか
- 実際にどのような影響があるか
- なぜ既存のテストでは防げないか
- どのように修正すべきか

可能性はあるが確認できないものは「追加調査項目」として分離する。

## 重要度

- **P0**: 重大なデータ損失、資金損失、全体停止など、即時対応が必要
- **P1**: 本番障害、シグナル誤算、主要機能の誤動作
- **P2**: 特定条件で発生する不具合、信頼性・性能・運用上の実害
- **P3**: 現在は動作するが、変更時の不具合や保守上の実害を生みやすい問題

## 出力形式

### 1. 結論
- **BLOCK**: リリースを止めるべき問題がある
- **CONDITIONAL PASS**: 条件付きで許容できる
- **PASS**: 確認範囲では重大な問題がない

### 2. レビュー対象と調査範囲
- 確認したディレクトリ
- 確認した主要な実行経路
- 実行したコマンドと結果
- 確認できなかった範囲

### 3. 問題一覧
重要度順に、発見した問題をすべて出力する。件数を制限しない。

各問題の形式:
```
#### [P1-001] 問題の短い名称
- ファイルと行番号:
- 発生条件:
- 影響:
- 根拠（コード引用）:
- なぜテストで防げないか:
- 修正方針:
- 確信度: 高／中／低
```

### 4. 追加調査項目
根拠不足で正式な問題と断定できない項目

### 5. カバレッジ表
各レビュー観点について「問題あり」「問題なし」「該当なし」「情報不足」を記載

### 6. 最終自己点検
提出前に以下を確認する:
- [ ] 主要なエントリーポイントをすべて確認したか
- [ ] 正常系と異常系の両方を確認したか
- [ ] 処理経路を追跡したか（ファイル単位ではない）
- [ ] ドメイン固有リスク（リーク・中立・数値安定性）を確認したか
- [ ] 2回目の反証レビューを実施したか
- [ ] 指摘ごとに具体的な根拠があるか
- [ ] 未確認範囲を隠していないか

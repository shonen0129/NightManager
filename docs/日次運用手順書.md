# 日次運用手順書

## 日米業種リードラグ市場中立戦略ファンド
**正式採用本番モデル:** Production Residual-BLPX-RA v2 (Residual-BLPX-RA v2)

本書は、日米業種リードラグ市場中立戦略ファンドの日次運用オペレーション、システム実行手順、安全監査基準、マニュアル・ロールバック手順、および監視・アラートのプロセスを定める実務マニュアルである。本書は『運用方針書』を補完する運用的詳細ドキュメントである。

---

## 1. 日次オペレーション手順

毎営業日、以下の手順に従ってシステムを実行し、注文発注およびポジション管理を行う。

### 1.1 米国市場クローズ後（日本時間早朝）の処理

1. **米国価格データの確定**
   米国市場クローズ後、当日の米国業種ETF15銘柄の終値を取得・確定させる。
2. **ターゲット残差化とデータの構築**
   過去60日のローリング OLS ベータに基づき、日本側ターゲットリターンを TOPIX 残差化した結合行列 $R^{\text{Residual-PCA}}_{\tau}$ を構築する。
3. **モデルの推定**
   直近250営業日のウィンドウに対して `winsor_sigma = 3.0` のウィンザライズを適用し、EWMA（半減期45営業日）で平均、標準偏差、相関行列を算出する。これをブロック分割し、正則化パラメータ $\alpha_{xx}=0.50, \alpha_{yx}=0.05, \alpha_{yy}=0.50$ を用いて $\Sigma_{XX}, \Sigma_{YX}, \Sigma_{YY}$ を算出する。
4. **BLPX予測行列およびシグナルの算出**
   Ridgeパラメータ $\rho=0.01$ を用いて基本予測行列 $B_{\text{blp}}$ を構成する。PCA正則化による $B_{\text{pca}}$、固定業種マッピング $M_{\text{sector}}$ による $B_{\text{sector}}$ を凸結合（$\lambda_{\text{pca}}=0.10, \lambda_{\text{sector}}=0.40$）して $B_{\text{struct}}$ を決定し、予測値 $\hat{z}_{J,t+1} = B_{\text{struct}} z_{U,t}$ を算出する。
5. **ギャップ調整済み予測分布の生成**
   以下のスクリプトを実行し、当日の gap 調整済み予測平均 $\mu_{\text{gap},j,t}$ および共分散行列 $\Omega_{\text{gap},t}$ を算出する。
   * **実行スクリプト**: `tools/production/compute_gap_adjusted_distribution.py`
   * **出力先**: `results/gap_adjusted_distribution/latest/matrices/`
6. **リスク調整スコアの算出と v1 フォールバックの確認**
   スコア $\text{score}_{j,t} = \mu_{\text{gap},j,t} / \sigma_{\text{gap},j,t}$ を算出する。gap data が欠損している場合は、自動的に v1 シグナルによるフォールバックに移行し、前日の v1 ウェイトを使用する（以降のステップ 7 をスキップ）。

### 1.2 翌朝（日本市場寄付き前後）の処理

7. **目標ウェイトの決定（8:45〜9:00）**
   リスク調整スコアに基づき、上位5銘柄をロング、下位5銘柄をショートとして選定する。`baseline_style` 加重で基本ウェイト $w'_{j,t+1}$ を算出し、PIT 三分位ビニング（RuleD）により動的グロス乗数 $m_t \in \{0.75, 1.00\}$ を適用して、最終目標ウェイト $w_{j,t+1} = w'_{j,t+1} \times m_t$ を決定する。
   * **実行スクリプト（v2本番）**: `tools/production/run_daily_production_v2.py`
   * **本番設定ファイル**: `configs/production/production.yaml` (v2用)
   * **v2特化設定**: `configs/production/production_v2_primary_ruleD.yaml`
   * **主要出力ファイル**:
     - `live/production_residual_blpx/latest_weights.csv`: 当日の最終目標ウェイト
     - `live/production_residual_blpx/v1_baseline_weights.csv`: v1目標ウェイト（フォールバック用）
     - `live/production_residual_blpx/production_scores.csv`: $\mu_{\text{gap}} / \sigma_{\text{gap}}$ スコア値
     - `live/production_residual_blpx/pit_binning.json`: PITビニングのパーセンタイル・閾値情報
     - `live/production_residual_blpx/production_audit.json`: 安全監査結果JSON
8. **自動安全監査（Safety Audit）の実行とフォールバック判断**
   目標ウェイトの算出と同時に、自動安全監査プログラムが走り、全監査項目を検証する（詳細は §2 参照）。監査が1項目でも失敗した場合、または致命的なデータ欠損が生じた場合は、自動的に以下のフォールバックルールを適用する。
   * **第1フォールバック**: 前日算出済みの Residual-BLPX v1 ウェイト（`v1_baseline_weights.csv`）を当日の取引ウェイトとして適用。
   * **最終フォールバック**: PCA-Ensemble (Sector Relative Ensemble) の算出フロー（Raw-PCA 50% / Residual-PCA 50% アンサンブル）に切り替え。
9. **リスク制限チェックと発注承認（9:00前）**
   ポートフォリオのネット／グロスエクスポージャー、参加率上限（ADVの10〜15%）をチェックする。グロスが200%（またはRuleDによる150%）の上限を超過している場合は、全銘柄ウェイトを同一比率で自動縮小する。VaR(99%)およびES(99%)が警戒水準を超過していないかを確認し、問題なければ発注を承認する。
10. **日本市場での発注・ポジション構築（9:10）**
    朝の寄付きから一定時間ノイズが落ち着いた「9:10」を基準時点として発注およびポジション構築を行う（gap_residualモードでは、9:00の寄付き価格確定後にギャップ補正シグナルを再計算し、9:10までに算出された最新シグナルに基づいて発注する）。

### 1.3 日中から大引け決済までの処理

11. **ポジション・モニタリング（日中）**
    保有ポジションの評価損益、および相場急変・サーキットブレーカー発動等の異常事態をリアルタイムで監視する。
12. **大引け決済（15:00〜15:30）**
    大引けにかけてポジションの反対売買（決済注文）を行い、15:30までに全ポジションをスクエアにする。オーバーナイトでのポジション保有は原則禁止する。
13. **実績照合およびレポート更新（16:00以降）**
    約定結果と注文目標の照合を行い、日次実績レポートを作成する。アルファ減衰モニタリング指標（ローリングIR、シグナルIC、クロスマーケット相関）を更新し、データベースに保存する。

## 2. 自動安全監査（Safety Audit）基準

日次実行フローにおいて、以下の 8 項目について自動検証を行い、結果を `production_audit.json` に記録する。`all_passed` フラグが `true` でない場合、注文執行を自動停止しフォールバックウェイトに切り替える。

1. **モデル名監査 (`model_name_audit`)**
   実行に使用されたモデル名が本番指定の `production_residual_blpx` に固定されているかを検証する。
2. **シグナル加重監査 (`signal_weight_audit`)**
   ウェイトの算出方式 `portfolio.weight_mode` が `signal`（確信度加重）であり、均等ウェイトになっていないことを検証する。
3. **日付整合監査 (`date_consistency_audit`)**
   シグナル算出日 `signal_date` が、実際の取引実行日 `trade_date` より1日以上過去であることを検証する（ルックアヘッドデータ防止）。
4. **残差化リーク監査 (`residual_leak_audit`)**
   OLSベータの推定において、当日のリターン情報が未来データとして学習ウィンドウに含まれていないこと（ベータが1日以上シフトして適用されていること）を検証する。
5. **コスト整合監査 (`cost_consistency_audit`)**
   パフォーマンス計算における純リターンが `net_return = gross_return - cost` の数式と完全に一致しているかを検証する。
6. **ウェイト制約監査 (`weight_constraint_audit`)**
   ポートフォリオがドルニュートラル（$\sum w_i = 0$）であり、無限大（`inf`）や欠損値（`NaN`）が含まれていないことを検証する。
7. **ティッカー順序監査 (`ticker_order_audit`)**
   入出力データの業種ベクトルの並び順が、規定の `JP_TICKERS` のインデックス順と完全に一致しているかを検証する。
8. **確信度分散監査 (`confidence_variance_audit`)**
   確信度調整に使用した条件付き共分散行列 $\Sigma_{Y|X}$ が正定値であり、算出された予測分散が有限かつ正（下限 $10^{-8}$ 以上）であることを検証する。

## 3. マニュアル・ロールバック手順

システムバグ、APIの接続障害、データ欠損、または自動安全監査の失敗時に、手動で前世代のモデルやフォールバックモデルに切り替える際の手順である。

### 3.1 v2 → v1 ロールバック手順
（gap 分布データ生成の失敗、または v2 モデル固有の不具合発生時）

1. **注文生成プロセスの停止**
   運用責任者の緊急承認のもと、ただちに自動発注スクリプトの実行プロセスを停止する。
2. **v1 ウェイトの確認**
   `live/production_residual_blpx/v1_baseline_weights.csv` が最新（前営業日深夜に正常算出された v1 ウェイト）であることを目視で確認する。
3. **ウェイトファイルの手動上書き**
   `v1_baseline_weights.csv` の内容をコピーし、当日発注システムが読み込む `latest_weights.csv` に手動で上書き保存する。
4. **設定ファイルの切り替え**
   `configs/production/production.yaml` を、バックアップされている v1 用設定ファイル（`configs/archive/production_residual_blpx_v1_before_v2_*.yaml` など）で上書き置換する。
5. **バージョンの確認**
   システムログを再起動し、ロードされたモデルバージョンが `production_residual_blpx_v1` に切り替わっていることを確認する。変更経緯を Git にコミットし、タグを付与する。

### 3.2 v1 → PCA-Ensemble ロールバック手順
（v1 モデルの監査失敗、またはBLPXモデル全体の致命的不具合発生時）

1. **注文生成プロセスの停止**
   ただちに自動発注スクリプトの実行を停止し、新規建て注文の送信をホールドする。
2. **設定ファイルの上書き**
   アクティブな本番設定ファイル `configs/production/production.yaml` を、バックアップされている旧PCA-Ensemble版設定ファイル（`configs/archive/production_before_sre.yaml` 等）で上書き置換する。
3. **プロセスの再起動と検証**
   実行システムを再起動し、ログ出力において読み込まれたモデル名が `sector_relative_ensemble` であることを確認する。
4. **記録と報告**
   ロールバック実行の経緯と変更履歴を Git で記録し、投資委員会およびリスク管理責任者にインシデント報告書を提出する。

## 4. 監視および緊急対応（アラート）手順

システムは日次の損失状況を監視し、以下の閾値に達した場合は定められたアクションを自動的または手動で実行する。

### 4.1 損失アラート基準とアクション

* **日次損失が NAV比 1.5% を超過（`daily_loss_warning`）**
  * **アクション**: システムは運用担当者へ高優先度の警告アラート（メール・チャット）を出力する。ポジションの異常値や急激なファクターの傾きがないかを確認する。
* **日次損失が NAV比 2.5% を超過（`daily_loss_stop`）**
  * **アクション**: 緊急運用停止を発動する。システムによる新規建て注文の生成・発注を自動停止し、既存ポジションについては大引け（または市場流動性がある可能な限り速やかなタイミング）で強制決済を実行する。
* **月次累積損失が NAV比 5.0% を超過（`monthly_loss_stop`）**
  * **アクション**: 戦略の運用を一時停止する。投資委員会を招集し、戦略パラメータの再検証、市場レジームの変化の有無、および運用再開に向けた移行計画を決定する。

### 4.2 VaR・ES の限界値超過時の対応

日次で計算される 1日VaR(99%) または 1日ES(99%) が以下の閾値を超過した場合は、リスク管理責任者に即時エスカレーションを行う。

| 指標 | 警戒水準（ポジション縮小検討） | 停止水準（新規建て停止） |
|:---|:---:|:---:|
| 1日VaR(99%) | NAV比 2.0% | NAV比 3.0% |
| 1日ES(99%) | NAV比 2.5% | NAV比 4.0% |

リスク管理責任者は、一時的なボラティリティ急上昇によるものか、ポートフォリオの集中によるものかを分析し、PMに対してグロスエクスポージャーの引き下げ（例: 200%から150%への強制縮小）を指示する。

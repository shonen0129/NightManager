# CLI実行スタック（停止）再発防止策

> 作成日: 2026-07-05
> 目的: CLI実行中にプロセスが停止・ハングする事象の共通原因を整理し、再発を防止する

---

## 1. スタックの共通パターン（5類型）

過去のスタック事象は、コードベース分析の結果、以下の5つのパターンに分類される。

### パターンA: 無限待機ループ（`wait_and_auto_close`）

**場所**: `src/leadlag/execution/close.py:298-306`

```python
while True:
    remaining = (target - datetime.now()).total_seconds()
    if remaining <= 0:
        break
    if remaining <= check_interval:
        time_module.sleep(remaining)
        break
    time_module.sleep(check_interval)  # 5分間スリープ
```

**症状**: `--auto-close` オプション使用時、14:50まで `while True` + `time.sleep(300)` で待機し続ける。launchd/cronから実行すると「止まった」ように見える。

**発生条件**: 朝9:05のdecision実行で `--auto-close` を付けた場合、5時間以上スリープする。

---

### パターンB: yfinanceダウンロードのハング

**場所**:
- `src/leadlag/data/fetcher.py:237-248`（標準mode）
- `src/leadlag/broker/tachibana/client.py:206-223`（Tachibana broker使用時のUS ETF取得）

```python
us_data = yf.download(US_TICKERS, start=..., end=..., auto_adjust=False)
```

**症状**: `yf.download()` にタイムアウトが設定されておらず、Yahoo Finance側のレート制限・サーバー遅延・メンテナンス時に無限待機になる。

**発生条件**: 標準mode（`--fast-mode`なし）でのdecision実行、またはTachibana broker使用時。

---

### パターンC: ファイルロックの競合

**場所**: `src/leadlag/data/cache.py:136-147`

```python
@contextlib.contextmanager
def exclusive_lock(lock_path: str):
    with open(lock_path, "a+b") as lock_file:
        _lock_file(lock_file, exclusive=True, non_blocking=False)  # ブロッキング
        ...
```

**症状**: `fcntl.flock(LOCK_EX)` はブロッキングモードで取得するため、別プロセスがロック保持中は無限待機する。前回実行のプロセスがクラッシュ・強制終了された場合、ロックファイルが残存し次回実行がハングする。

**発生条件**: decision と close が同時実行された場合、または前回実行が異常終了した場合。

---

### パターンD: API再試行の指数バックオフ

**場所**: `src/leadlag/broker/kabu/api.py:268-311`

```python
wait_time = backoff_factor * (2**attempt)
time.sleep(wait_time)  # 指数バックオフ
```

**症状**: kabuステーション API が応答しない場合、リトライごとにスリープ時間が指数増大（1s → 2s → 4s → 8s ...）。`max_retries`回まで繰り返すため、合計で数十秒〜数分待機する。

**発生条件**: kabuステーション API が起動していない、またはネットワーク到達不可。

---

### パターンE: 注文後のフィル確認待ち

**場所**: `src/leadlag/execution/helpers.py:806`

```python
time.sleep(wait_seconds)  # 約定確認まで待機
```

**症状**: 注文送信後、約定価格を取得するために `wait_seconds` 秒スリープする。デフォルト値が長い場合、プロセスが停止しているように見える。

**発生条件**: Tachibana broker で注文送信後のフィル確認時。

---

## 2. スタック検出の難しさ

| 課題 | 説明 |
|------|------|
| **ログ出力なし** | `time.sleep` 中はログが出力されず、生存確認が困難 |
| **シグナル処理なし** | SIGTERM/SIGINT をトラップしていないため、launchd からの停止要求が無視される |
| **タイムアウト未設定** | yfinance呼び出しにタイムアウトがない |
| **ロック未クリーンアップ** | クラッシュ時のロックファイル残存対策なし |

---

## 3. 再発防止策

### 3.1 即座適用すべき対策（高優先度）

#### [P1] `wait_and_auto_close` の分離

`--auto-close` による同一プロセス内待機を廃止し、close を別プロセス（既存のlaunchd `com.leadlag.close`）に任せる。

- **対応**: バッチスクリプト `run_decision.sh` から `--auto-close` フラグを削除
- **理由**: 既に `com.leadlag.close.plist` が14:50にcloseを実行する仕組みがあるため、decision内での待機は冗長

#### [P2] yfinance にタイムアウト導入

`yf.download()` を `signal.alarm` または `concurrent.futures.ThreadPoolExecutor` + `future.result(timeout=60)` でラップし、60秒でタイムアウトさせる。

- **対象**: `src/leadlag/data/fetcher.py` の `download_data()` および `update_incremental()`
- **対象**: `src/leadlag/broker/tachibana/client.py` の `fetch_us_etf_returns()`

#### [P3] ファイルロックにタイムアウト追加

`exclusive_lock` にタイムアウト引数を追加し、規定時間内にロック取得できない場合は例外を送出する。

```python
@contextlib.contextmanager
def exclusive_lock(lock_path: str, timeout: float = 30.0):
    # タイムアウト付きでロック取得
    # 取得できない場合は LockTimeoutError を送出
```

- **対象**: `src/leadlag/data/cache.py`
- **呼び出し元**: `cli.py:296`, `fast.py:461`, `cache.py:300`

#### [P4] API再試行の最大待機時間上限設定

指数バックオフの最大スリープ時間を設定し、全体の再試行時間に上限を設ける。

- **現在**: `backoff_factor * (2**attempt)` で無限増大
- **改善**: `min(backoff_factor * (2**attempt), max_sleep)` を適用（`max_sleep=10`）
- **対象**: `src/leadlag/broker/kabu/api.py`

### 3.2 中期対策（中優先度）

#### [P5] スタック検出用ハートビートログ

長時間の `time.sleep` 前後にログを出力し、外部からプロセスの生存確認を可能にする。

```python
logger.info("[HEARTBEAT] Sleeping for %.0f seconds until auto-close time %s", wait_seconds, auto_close_time)
```

- **対象**: `close.py` の `wait_and_auto_close`、`helpers.py` のフィル確認待ち

#### [P6] シグナルハンドラ導入

SIGTERM/SIGINT をトラップし、実行中のリソース（API接続、ファイルロック）をクリーンアップして終了する。

```python
import signal

def _cleanup_handler(signum, frame):
    logger.warning("Received signal %d, shutting down gracefully...", signum)
    raise SystemExit(128 + signum)

signal.signal(signal.SIGTERM, _cleanup_handler)
```

- **対象**: `src/leadlag/cli.py` の `main()`

#### [P7] Tachibana broker の yfinance依存除去

Tachibana broker の `fetch_us_etf_returns()` が yfinance に依存しているため、US ETFデータをローカルキャッシュから取得する方式に切り替える。

- **対象**: `src/leadlag/broker/tachibana/client.py:197-233`
- **代替案**: `fetch_us_returns_from_api()` (fast.py) と同様にローカルJSONキャッシュを使用

### 3.3 長期対策（低優先度）

#### [P8] プロセス監視の導入

launchd の `Watchdog` 設定または外部監視スクリプトで、CLI実行プロセスの実行時間を監視し、規定時間を超えた場合は強制終了＋通知する。

#### [P9] スタック時の自動ダンプ

長時間スリープに入る前にスレッドダンプまたはプロセス状態をファイルに出力し、事後分析を可能にする。

## 5. 確認チェックリスト（運用時）

CLI実行前に以下を確認することで、スタックの大部分を未然に防げる：

- [ ] **broker API起動確認**: kabuステーション または 立花証券アプリが起動しているか
- [ ] **`--auto-close` 未使用確認**: decision実行に `--auto-close` が付いていないか（closeは別プロセスで実行）
- [ ] **前回プロセス残存確認**: `ps aux | grep leadlag` で前回プロセスが残っていないか
- [ ] **ロックファイル確認**: `results/.cache/*.lock` に古いロックファイルが残っていないか
- [ ] **ネットワーク確認**: Yahoo Finance / Google Finance に到達可能か
- [ ] **`--fast-mode` 使用確認**: 標準mode（yfinance使用）を避け、fast-mode（API経由）を使用しているか

---

## 6. スタック発生時の復旧手順

```bash
# 1. プロセス確認・強制終了
ps aux | grep leadlag
kill -9 <PID>

# 2. ロックファイル削除
rm -f results/.cache/*.lock

# 3. ログ確認
tail -50 logs/decision_*.log

# 4. 手動再実行（fast-mode推奨）
PYTHONPATH=src .venv-mac/bin/python -m leadlag.cli decision \
    --api-enable --fast-mode --capital-from-wallet --text-output
```
